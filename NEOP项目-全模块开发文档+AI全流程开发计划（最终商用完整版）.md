## 一、全局通用强制规则（AI 100%严格执行）

### 1\.1 业务核心约束

- 全程**禁止余额钱包、禁止资金池、禁止余额流转**，所有现金奖励绑定单个任务工单，独立结算、独立提现、独立打款日志

- 所有用户数据严格隔离，用户仅可查看自身数据，后台拥有全部权限

- 所有状态流转**禁止跨状态更新**，必须按既定流程执行

- 所有接口必须做登录拦截、权限校验、参数非空校验

- 所有图片上传统一复用系统OSS/本地上传工具类，多图采用英文逗号分隔存储

### 1\.2 通用状态规范

- 通用启用禁用：0=禁用、1=启用

- 通用删除逻辑：全部采用**逻辑删除**，不物理删除数据

- 时间字段：统一使用datetime，默认CURRENT\_TIMESTAMP，更新字段自动更新

- 主键规则：所有表主键统一bigint自增，关联字段统一对应主键类型

### 1\.3 代码开发规范 \&amp; 项目包结构定义（AI强制遵守）

- **包结构强制统一（NEOP后端根包：com\.gongziyu\.neop）**

    - com\.gongziyu\.neop\.controller —— 控制器接口层

    - com\.gongziyu\.neop\.service —— 业务逻辑层

    - com\.gongziyu\.neop\.service\.impl —— 业务实现层

    - com\.gongziyu\.neop\.mapper —— 数据库映射层

    - com\.gongziyu\.neop\.entity —— 数据库实体

    - com\.gongziyu\.neop\.config —— 全局配置类

    - com\.gongziyu\.neop\.util —— 工具类（微信、加密、通用）

    - com\.gongziyu\.neop\.task —— 定时任务

    - com\.gongziyu\.neop\.exception —— 全局异常处理

    - com\.gongziyu\.neop\.common —— 公共返回、常量、分页

- 数据库字段下划线命名，实体类小驼峰命名，严格对应无偏差

- 所有接口统一返回**Result通用封装类**，固定结构不可改动

- 分页接口统一接收page、limit参数，统一返回分页数据结构

- 定时任务仅执行数据状态更新、过期处理，不处理复杂业务逻辑

- 图片存储规则：前端传图片URL数组，**后端统一转为英文逗号分隔字符串入库**，查询时后端自动拆分返回数组给前端

**1）普通接口统一返回格式**

```json

  "msg": "success",
```

**2）分页接口统一返回格式**

```json

  "code": 200,
  "msg": "success",
  "data": {
    "total": 120,

    "current": 1,
    "size": 10,

  }
```

## 三、各模块数据表结构\+字段关联说明

### 3\.1 系统权限模块（基础支撑模块）

**核心作用**：后台账号权限、菜单路由、系统全局配置，支撑整个NEOP系统后台运行

**表关联关系**：

- sys\_admin（管理员）→ sys\_admin\_role（关联角色ID）→ sys\_role（角色）→ sys\_role\_menu（关联菜单ID）→ sys\_menu（菜单权限）

- sys\_dict：全局数据字典，供所有模块下拉、状态展示使用

- sys\_config：系统全局参数配置（支付参数、上传参数、任务默认参数等）

### 3\.2 用户基础模块

**核心作用**：前台用户账号体系、微信授权、收货地址管理，所有业务模块的用户载体

**表关联关系**：

- user\_info（用户主表）：所有用户数据唯一主体，所有业务表通过user\_id关联

- user\_wechat：一对一关联user\_info，存储用户微信openid/unionid，为任务提现、微信登录提供参数

- user\_address：一对多关联user\_info，一个用户多个收货地址，支撑电商订单收货场景

### 3\.3 营销增值模块

**核心作用**：会员充值、积分签到、用户邀请分销，提升用户留存与裂变

**表关联关系**：

- member\_package（会员套餐）：后台配置充值套餐，无用户关联，全局通用

- member\_user：一对一关联user\_info，记录用户会员开通、到期、续费信息

- point\_user：一对一关联user\_info，记录用户积分总额、剩余**可用积分**

- point\_log：多对一关联user\_info，所有积分变动流水记录，**balance字段=变动后可用积分**

- invite\_user：一对一关联user\_info，记录用户邀请码、上下级关系

- invite\_reward\_log：多对一关联user\_info，记录邀请奖励发放记录，**仅支持积分奖励，无余额奖励**

### 3\.4 电商交易模块

**核心作用**：商品上架、购物、下单、支付、订单管理，完整电商交易闭环

**表关联关系**：

- product\_category（商品分类）：父级嵌套结构，关联下级分类、商品

- product（商品）：多对一关联分类表，存储商品核心信息

- cart（购物车）：多对一关联用户、商品，存储用户选购商品

- pay\_order：多对一关联用户，**仅用于会员充值订单**，彻底移除余额充值业务逻辑

- order（订单主表）：多对一关联用户，存储订单核心信息

- order\_product（订单明细）：多对一关联订单、用户、商品，记录下单商品详情

### 3\.5 任务打款模块（核心特色模块）

**核心业务流程**：后台发布任务→用户领取→提交审核→后台审核通过→后台授权打款→用户解锁提现→单笔微信打款到零钱

**表关联关系**：

- task\_info（任务主表）：后台发布的所有任务，全局通用模板

- task\_user\_receive（用户任务领取记录）：核心中间表，关联用户、任务，存储**审核状态、授权状态、提现状态**三大核心状态

- task\_submit（任务工单）：一对一关联领取记录，存储用户提交的截图、备注、审核结果

- task\_pay\_log（打款日志）：一对一关联领取记录，存储每笔任务提现、微信打款完整记录

### 3\.6 数据统计模块

**核心作用**：每日自动统计用户注册、活跃、订单营收数据，供后台数据看板展示

**表关联关系**：无外键关联，每日独立生成统计数据，唯一约束统计日期

## 四、核心模块接口文档（全量完整版）

### 4\.1 系统\&amp;用户\&amp;电商\&amp;营销通用接口（简述）

包含后台登录、权限校验、用户注册登录、会员充值、积分签到、邀请裂变、商品管理、购物车、订单支付、地址管理等基础接口，沿用现有成熟CRUD逻辑，严格匹配数据表字段，状态流转遵循通用规范。

### 4\.2 任务模块全套接口（核心重点）

#### 前台用户端接口

**1\. 获取任务列表**

接口：GET /api/task/list

参数：page、limit

返回：任务封面、标题、简介、奖励金额、剩余数量、排序权重

筛选规则：仅返回状态=1上架、未过期、有量的任务

**2\. 任务图文详情**

接口：GET /api/task/info/\{taskId\}

返回：任务全部信息、图文步骤、奖励金额、有效期、领取限制

**3\. 领取任务**

接口：POST /api/task/receive

参数：taskId

业务规则：校验重复领取、每日限量、总量限量；自动读取用户微信openid存入记录；自动生成任务过期时间

**4\. 提交任务审核材料**

接口：POST /api/task/submit

参数：receiveId、submitImages、submitNote

业务规则：前端提交图片URL数组，后端接收后自动拼接为英文逗号分隔字符串入库；仅可提交一次；提交后状态由「待提交」变为「待审核」

**请求示例JSON**

```json
{
  "receiveId": 10001,
  "submitImages": [
    "https://xxx.com/1.png",
    "https://xxx.com/2.png"
  ],
  "submitNote": "已按要求完成任务步骤"
}
```

**响应示例JSON**

```json
{
  "code": 200,
  "msg": "success",
  "data": true
}
```

**5\. 我的任务列表**

参数：status（0全部、1待提交、2待审核、3已通过、4已驳回、5已过期）

返回：任务信息、审核状态、授权打款状态、提现状态

**6\. 我的任务详情**

接口：GET /api/task/my/info/\{receiveId\}

**7\. 任务单笔提现（核心）**

接口：POST /api/task/withdraw/apply

参数：receiveId

三层强校验：状态=3审核通过、grant\_pay=1已授权、withdraw\_status=0未提现

执行逻辑：生成提现单号→更新提现状态为「已申请提现/处理中」→调用微信企业付款→根据同步结果初步更新状态→等待异步回调最终确认成功/失败、写入完整打款日志

**提现申请请求示例**

```json
{
  "receiveId": 10001
}
```

**提现申请响应示例**

```json
{
  "code": 200,
  "msg": "提现申请提交成功，等待打款",
  "data": {
    "tradeNo": "NEOP-TASK-20260426120001",
```

**8\. 我的提现记录**

接口：GET /api/task/withdraw/log

#### 后台管理端接口

### 阶段5\.8 微信企业付款技术完整实现（AI强制严格开发）

**核心说明**：微信企业付款到零钱为异步\+同步双校验接口，必须完整实现证书加载、签名、下单、回调、失败重试、补发逻辑。

**1\. 对接所需配置参数（存入sys\_config系统配置）**

- 微信商户号 mch\_id

- 商户API密钥 api\_key

- 商户API证书 apiclient\_cert\.p12（放置resource/cert目录）

- 微信支付异步回调域名（统一配置为：https://neop\.gongziyu\.com/api/wechat/pay/callback）

**2\. 技术实现细节**

- 证书处理：后端工具类启动预加载P12证书，全局单例调用，禁止重复加载

- 签名算法：采用微信MD5签名规则，参数排序、加密、验签

- 同步调用：调用企业付款接口，获取微信同步返回结果（仅代表请求成功，不代表到账）

- 异步回调接口：新增 `/api/wechat/pay/callback` 接收微信异步通知，最终更新打款状态

**3\. 打款状态最终判定逻辑**

- 同步调用成功 → pay\_status=1 处理中

- 异步回调成功 → pay\_status=2 打款成功，更新pay\_time

- 回调返回失败 → pay\_status=3 打款失败，记录fail\_reason

**4\. 失败处理 \&amp; 重试 \&amp; 补发机制**

- 自动重试：打款处理中超过10分钟未回调，定时任务自动重试1次

- 后台手动补发：后台打款记录列表增加【手动补发】按钮，仅失败订单可操作

- 所有重试、补发行为写入日志，可追溯

**5\. 回调接口安全规则**

- 回调必须验签，非法请求直接拦截

- 防止重复回调，同一tradeNo仅处理一次

- 数据更新加事务，保证状态一致性

## 六、AI开发强制禁忌与校验标准

### 6\.1 禁止操作

- 禁止新增任何用户余额、资金池相关字段与逻辑，彻底废弃所有余额充值、余额奖励相关业务

- 禁止跳过三层提现校验、禁止跨状态更新任务数据
