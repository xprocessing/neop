### 3.2 二十七章：代码生成器模板（提高开发效率）

**问题**：如果让AI从零写28张表的Entity/Mapper/Service，效率低下且容易出错。

**解决方案**：提供BaseEntity和BaseController的完整代码模板，要求AI使用MyBatis-Plus代码生成器。

**BaseEntity模板**：
```java
// 所有表的公共字段（id/deleted/createTime/updateTime）
@Data
public class BaseEntity {
    @TableId(type = IdType.AUTO)
    private Long id;
    
    @TableLogic
    private Integer deleted;
    
    @TableField(fill = FieldFill.INSERT)
    private Date createTime;
    
    @TableField(fill = FieldFill.INSERT_UPDATE)
    private Date updateTime;
}
```

**使用代码生成器**：AI只需运行一次生成器，即可生成28张表的Entity/Mapper/Service代码，然后人工调整。

### 3.3 二十八章：JWT Token代码（防止鉴权漏洞）

**问题**：文档第10章提到了Token，但没给具体生成方法。AI可能会写出有漏洞的鉴权代码。

**解决方案**：提供完整的JwtUtil工具类代码，明确Token生成时必须放入的Claims，禁止放入敏感字段。

**关键代码**：
```java
// 生成移动端Token（必须放入的Claims）
Map<String, Object> claims = new HashMap<>();
claims.put("userId", userId);
claims.put("type", "mobile");
claims.put("nickname", nickname);
// 注意：禁止放入password、openid等敏感字段

String token = Jwts.builder()
    .setClaims(claims)
    .setExpiration(new Date(System.currentTimeMillis() + mobileExpire))
    .signWith(getSignKey(), SignatureAlgorithm.HS256)
    .compact();
```

### 3.4 二十九章：增删改查标准范式（确保代码一致性）

**问题**：AI全自动开发时，最容易出现的问题就是"写后面忘前面"，代码风格不一致。

**解决方案**：定义一个**标准施工范式**，要求AI所有模块都按这个分层来写。

**标准范式**：
```java
// 1. Controller层 - 只做参数校验和结果返回
@RestController
@RequestMapping("/api/admin/task")
public class TaskController extends BaseController {
    @PostMapping("/save")
    public Result<Void> save(@Valid @RequestBody TaskSaveDTO saveDTO) {
        taskService.save(saveDTO);
        return success(null);
    }
}

// 2. Service接口定义
public interface TaskService extends IService<TaskInfo> {
    void save(TaskSaveDTO saveDTO);
}

// 3. ServiceImpl - 业务逻辑层
@Service
public class TaskServiceImpl extends ServiceImpl<TaskInfoMapper, TaskInfo> implements TaskService {
    @Transactional(rollbackFor = Exception.class)
    public void save(TaskSaveDTO saveDTO) {
        TaskInfo taskInfo = BeanUtil.copyProperties(saveDTO, TaskInfo.class);
        taskInfoMapper.insert(taskInfo);
    }
}

// 4. Mapper层 - 简单SQL用MyBatis-Plus，复杂SQL写XML
@Mapper
public interface TaskInfoMapper extends BaseMapper<TaskInfo> {
}
```

**AI指令**："以任务管理为例，定义标准施工范式，后续所有模块（会员、电商、商品分类）严格遵循此模板，只变更实体名和字段名。"

**问题**：代码写出来不是结束，能跑通才是。AI开发完成后，往往缺少测试验证。

**解决方案**：强制要求AI完成每个阶段的开发后，必须提供可导入Postman/ApiFox的JSON接口集合。

| **异常流** | 参数缺失/Token过期 | 返回对应错误码（401/400） |
| **边界流** | 重复支付、重复签到、库存边界 | 返回429或业务错误码 |

### 3.7 三十二章：Flyway增量脚本规范（保护数据库版本控制）

**问题**：对于AI来说，它很难在开发过程中完美维护增量脚本。AI可能会修改已存在的Flyway SQL文件，导致数据库版本控制失效。

**解决方案**：明确禁止修改已存在的Flyway SQL文件，任何变更都必须新建增量脚本。

**核心规则**：
```
在开发过程中，**严禁修改已存在的Flyway SQL文件**。
任何表结构或数据的变更，都必须新建一个`V{下一版本号}__{描述}.sql`文件。

示例：
- 初始建表：V1.0.0__init_schema.sql
- 新增字段：V1.0.1__add_task_info_reward_point.sql
- 修改字段：V1.0.2__modify_product_price_type.sql
```

**AI开发流程规范**：
```
1. 创建功能分支：git checkout -b feature/任务模块
2. 编写代码：生成Entity/Service/Controller
3. 如需要修改数据库：
   a. 创建新的Flyway增量脚本（V1.0.1__xxx.sql）
   b. 绝不修改已有的Flyway脚本
4. 本地测试：运行项目，Flyway自动执行增量脚本
5. 编写测试用例：Postman集合 + 单元测试
6. 提交代码：git commit -m "feat: 新增任务管理模块"
7. 合并到开发分支
```
